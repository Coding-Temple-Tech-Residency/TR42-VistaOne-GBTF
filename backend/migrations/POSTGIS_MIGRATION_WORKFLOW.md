# Alembic/PostGIS Migration Workflow

## Problem
Alembic autogenerate may try to drop or create the `spatial_ref_sys` table, which is managed by the PostGIS extension and should never be altered by migrations. This causes migration failures and breaks the database.

## Workflow
1. **After running `flask db migrate` (or Alembic autogenerate):**
   - Open the new migration script in `backend/migrations/versions/`.
   - Search for any operation on `spatial_ref_sys` (e.g., `op.drop_table('spatial_ref_sys')`, `op.create_table('spatial_ref_sys')`).
   - Comment out or remove these lines. Add a comment:
     ```python
     # op.drop_table('spatial_ref_sys')  # Skipped: required by PostGIS extension, cannot be dropped independently
     ```
2. **Before running `flask db upgrade`:**
   - Confirm that no migration script attempts to drop or create `spatial_ref_sys`.
3. **(Optional) Add a pre-commit hook or CI check** to scan migration scripts for forbidden operations and warn if found.
4. **Document this process** in the README and share with your team.

## Why?
- `spatial_ref_sys` is a system table required by PostGIS. Dropping or recreating it will break spatial features and may corrupt your database.
- Alembic does not know about PostGIS system tables, so it may try to remove them if not present in your models.

## Reference
- Always review autogenerated migrations for PostGIS system table operations.
- Never allow a migration to drop or create `spatial_ref_sys`.
